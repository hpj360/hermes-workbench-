## 二、Skill 编写精华（腾讯技术工程）

### 2.1 Skill 的本质

Skill = 结构化的 Prompt Engineering，通过标准格式把：
- **指令**（按什么步骤干活）
- **上下文**（项目背景、团队规范）
- **工具**（脚本、配置模板）

打包成 AI 可理解、可执行的指令集。类比：裸 AI = 刚入职的新人啥都得问；装了 Skill = 拿到了老员工的操作手册。

### 2.2 SKILL.md 标准结构

```
---
name: skill-name              # 唯一标识（小写+连字符）
description: 何时使用...       # 触发条件 + 触发关键词
compatibility: 依赖环境...    # 运行环境要求
allowed-tools: Bash(python:*)  # 授权工具
---

# 技能名称

## 角色
你是一名...

## 核心流程
1. 确认...
2. 执行...
3. 输出...

## 输出格式
| 字段 | 值 |
|---|---|
...

## 异常处理
- 若失败...
- 若超时...

## 示例对话
用户：...
助手：（执行...）
```

### 2.3 Skill 五大设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **单一职责** | 一个 Skill 只做一件事 | 一个 Skill 同时做"查数据+写Excel+发邮件" |
| **输入输出标准化** | 统一 JSON 入参/出参 | 参数格式随意，无类型约束 |
| **全链路异常兜底** | 网络超时/参数错误/限流预设返回 | 无异常处理，Agent 崩溃 |
| **自描述原则** | 自然语言注释说明用途和参数 | 只有代码无注释 |
| **无状态设计** | 单次调用不留存临时数据 | 在 Skill 内维护状态 |

### 2.4 Skill 反模式（避坑清单）

| 反模式 | 危害 | 正确做法 |
|--------|------|---------|
| 指令塞太多 | 信息过载，AI "晕掉" | 正文不超过 500 行，详情放 references/ |
| 忽略安全校验 | 危险命令被 AI 执行 | 路径白名单校验，禁止直接拼接用户输入 |
| 日志污染 stdout | JSON 解析失败 | 日志用 `sys.stderr`，stdout 只输出 JSON |
| 忽略验证模型 | Skill 自评不客观 | 独立验证 Skill 检查输出质量 |
| 触发条件模糊 | 该用时不用，不该用时乱用 | description 写清楚"何时用" |

## 三、Loop Engineering 精华（卡兹克）

### 3.1 四次跃迁的完整叙事

```
Prompt Engineering → Context Engineering → Harness Engineering → Loop Engineering
好好说话              给够信息             设规则约束          让系统自己跑
核心能力：语言表达     核心能力：信息组织     核心能力：规则制定     核心能力：目标定义
```

**卡兹克的洞察**：四次跃迁对应四门古老学科——语言学、信息科学、控制论、**管理学**。Loop Engineering 的核心竞争力不在工程，在管理。

### 3.2 Loop 的五个组件（Addy Osmani 框架 + 卡兹克修正）

| # | 组件 | 作用 | 常见误区 |
|---|------|------|---------|
| 1 | **定时任务**（心跳） | 自动启动循环，定时/事件触发 | 没有定时 = 还得手动踢 = 不是 Loop |
| 2 | **工作树隔离**（Worktree） | 每个子 Agent 独立空间，避免冲突 | 多个 Agent 改同一个文件 = 合并地狱 |
| 3 | **知识管理体系** | 跨会话记忆 + CLAUDE.md + 文档路由 | 过期信息 + 膨胀的 CLAUDE.md = 加速做错 |
| 4 | **连接器**（MCP） | 给 Agent 装上手脚，接 GitHub/数据库等 | 只能看文件系统 = 能力极其有限 |
| 5 | **子 Agent 分离** | 做事和检查分开，writer/judge 分离 | 自己写代码自己评分 = 永远觉得自己做对了 |

**卡兹克修正**：第 3 项 Addy Osmani 写的是"skill"，但卡兹克认为单 skill 不够，必须是**知识管理体系**（CLAUDE.md + 记忆 + docs + 定期梳理 Skill 如"洁癖.skill"）。

### 3.3 目标定义：Loop 的灵魂

**同一个工具，同一个模型，区别只在于目标定义得好不好。**

| 目标定义 | 结果 |
|---------|------|
| "把这个应用优化一下" | Claude 不知道什么叫"好了"，可能改了一点就停，或改到面目全非 |
| "test 目录下所有测试通过，tsc --noEmit 零报错，npm run lint 零违规" | 每轮自动跑三个命令，全过了就停，清清楚楚 |

**目标定义的四步框架**（卡兹克经验）：
1. 完成标准要可以被**机器验证**
2. 边界条件要跟完成标准**一起定义**（不能怎么做）
3. 要有**失败的降级方案**
4. 目标要**分层**（全局约束 vs 当前任务目标）

### 3.4 古德哈特定律：最阴险的陷阱

> **当一个衡量指标变成了目标本身，它就不再是一个好的衡量指标。**

**例子**：你的 Loop 条件是"测试全部通过" → Agent 直接把失败的测试删了 → 条件满足但真正的问题没解决。

**解法**：Harness（约束）是 Loop（驱动力）的另一半。Harness 告诉 Agent "可以这样做，但这条线不能越"。

```
Loop = 驱动力（往那个方向一直跑）
Harness = 约束（划定边界，不能怎么做）
两者相加 = 完整系统
```

### 3.5 管理学视角：管 Agent 比管人还难

| 管人 | 管 Agent |
|------|---------|
| 人可以理解模糊意图 | Agent 按自己的理解执行，不一定会主动确认 |
| 人会说"需求不太清楚" | Agent 会自信地做错并自信地报告"完成了" |
| 人可以容忍一定的模糊 | Agent 对模糊的容忍度为零，模糊 = 随机结果 |

## 四、增量知识：哪些是新沉淀

### 4.1 与已沉淀知识的对比

| 已沉淀知识 | 本文补充/强化 |
|-----------|-------------|
| Harness 五组件（指令/约束/反馈/记忆/编排） | Loop 五组件与 Harness 的**组合关系** |
| Skill 做小原则 | Skill 反模式（5 条具体避坑） |
| JSON 结构化输出规范 | 加了一条：**日志用 stderr，stdout 只输出 JSON** |
| — | **古德哈特定律在 Agent 场景的放大效应** |
| — | **目标定义四步框架**（可直接套用） |
| — | **管 Agent 的逻辑 = 管人的逻辑，但要求更高** |
| — | **四门古老学科视角**（语言学/信息科学/控制论/管理学） |

### 4.2 两条金句沉淀

1. **"对人如此，对 AI 也如此。你能不能把一个模糊的意图，翻译成一组可衡量、可验证的完成条件。"**
   — 管人的逻辑 = 管 Agent 的逻辑，核心能力都是目标定义

2. **"一个好的目标定义，不能只有做完了的标准，还必须有不能怎么做的边界。"**
   — Loop（驱动力）+ Harness（约束）= 完整系统

### 4.3 可直接复用的工具

| 工具 | 来自 | 用途 |
|------|------|------|
| Skill 反模式检查清单 | 腾讯技术工程 | 写完 Skill 后逐条自检 |
| 目标定义四步框架 | 卡兹克 | 定义任何自动化任务的目标 |
| 古德哈特定律检测 | 卡兹克 | 检查目标定义是否会被钻空子 |

## 五、落地建议

### 5.1 对 Skill 编写者的建议

1. 写 Skill 前先过一遍"反模式检查清单"（5 条）
2. description 写清楚触发关键词，这是 AI 判断"何时用"的唯一依据
3. 正文不超过 500 行，详情放 references/
4. 所有脚本输出必须包含 `success` 布尔字段
5. 日志用 `sys.stderr`，stdout 只输出 JSON

### 5.2 对 Loop 构建者的建议

1. **先定义目标，再搭 Loop**。目标不清，整个自动化链条 = 一坨狗屎
2. 每个目标同时定义"完成标准"和"不能怎么做的边界"
3. 设计验证机制（writer/judge 分离），不能让 Agent 自己给自己打分
4. 知识管理体系必须干净（定期用"洁癖 Skill"梳理）
5. 永远保留一个能说"不行"的东西：测试、类型检查、真实报错

### 5.3 从 Skill 到 Loop 的进化路径

```
L1: 写好单个 Skill（腾讯技术工程）
   ↓
L2: 把多个 Skill 按工作流编排（Harness Engineering）
   ↓
L3: 给 Harness 装上定时器和验证循环（Loop Engineering）
   ↓
L4: 定义清晰的目标 + 干净的护栏 = 放手的系统
```

**技能习得顺序**：先学会写 Skill（执行层），再学会定义目标（控制层）。倒过来走容易掉进"模糊目标 + 无限循环 = 烧钱"的大坑。
